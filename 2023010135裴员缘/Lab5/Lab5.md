# Lab5：IP 与以太网的包收发操作

## 实验背景

本实验围绕 IP 模块与以太网在包收发过程中的角色展开，重点观察以下内容：

1. 网络包的基本结构：头部（IP 头部 + MAC 头部）与数据
2. IP 头部各字段的含义：版本号、TTL、协议号、发送方/接收方 IP 地址等
3. MAC 头部各字段的含义：接收方/发送方 MAC 地址、以太类型
4. IP 地址与 MAC 地址的区别与协作
5. ARP 协议如何通过 IP 地址查询 MAC 地址
6. 路由表的结构与查询方式
7. UDP 协议与 TCP 协议的区别：无连接、无确认、无重传
8. UDP 头部结构：发送方端口号、接收方端口号、数据长度、校验和
9. ICMP 协议的作用与常见消息类型（Echo、Destination Unreachable 等）

2. TTL 字段的作用是什么？如果 TTL 降为 0 会发生什么？

4. IP 头部中的源 IP 地址和目的 IP 地址分别是谁的地址？它们与 MAC 头部中的源/目的 MAC 地址有什么区别？
IP 头部的源 IP 和目的 IP 是发送方和接收方主机的 IP 地址，用于在网络层标识通信端点，实现跨网段寻址；MAC 头部的源 / 目的 MAC 地址是当前链路中发送方和下一跳设备（或接收方）的物理地址，仅用于同一局域网内的帧传输，每次转发 MAC 地址都会更新，而 IP 地址在传输过程中保持不变

2. DNS 服务器的 IP 通常是外网地址。本任务中目的 MAC 地址是 DNS 服务器的 MAC 地址还是你本机网关（路由器）的 MAC 地址？为什么？

以太网帧中需要同时使用 IP 地址和 MAC 地址，二者缺一不可。IP 地址负责跨网段的端到端寻址，让数据能从源主机路由到目标主机；MAC 地址负责同一网段内的链路层传输，让数据能在每一跳设备之间准确传递。只用 IP 地址无法完成局域网内的直接交付，只用 MAC 地址无法实现跨网段路由，因此必须结合使用。

### 任务五：观察 ARP 协议

ARP（Address Resolution Protocol，地址解析协议）用于根据 IP 地址查询 MAC 地址。只要计算机处于联网状态，Wireshark 中通常会持续出现 ARP 包（邻居发现、缓存刷新等），可以直接观察。如果抓包一段时间后仍未看到 ARP 包，再手动触发。

**第一步：在 Wireshark 中设置过滤器**

```text
arp
```

**第二步：在包列表中找 ARP 包**

正常联网的设备每隔几分钟就会自动发送 ARP 请求，等待即可。如果等了一会儿仍没有，可以选择以下任一方式手动触发：

- **方式 A（推荐）**：在终端中执行 `arping`

  ```bash
  # Linux（通常已预装）
  sudo arping -c 3 <网关IP>

  # macOS（如果没有，先执行：brew install arping）
  sudo arping -c 3 <网关IP>

  # Windows（可从 https://github.com/ThomasHabets/arping/releases 下载）
  arping -c 3 <网关IP>
  ```

- **方式 B**：先清除 ARP 缓存，再 ping 同网段地址

  ```bash
  # 清除 ARP 缓存
  # Linux:   sudo ip neigh flush all
  # macOS:   sudo arp -d -a
  # Windows: arp -d *

  # 然后 ping 网关
  ping <网关IP> -c 2
  ```

> **注意**：如果目标是 `127.0.0.1` 或外网地址，ARP 不会出现。回环接口不经过以太网，外网地址的 MAC 地址是路由器的（通常已缓存）。

**第三步：点击 ARP 请求包（Opcode 为 request），展开详情**

**第四步：填写下表**

| 项目                     | 你的填写内容 |
| :----------------------- | :----------- |
| ARP 请求的目的 MAC 地址 |ff:ff:ff:ff:ff:ff|
| ARP 请求中查询的目标 IP |192.168.96.28|
| ARP 响应中返回的 MAC 地址 |68:7a:64:72:d2:92|
| 该 ARP 包是自动出现还是手动触发的 |自动出现|

简答题：

1. ARP 请求的目的 MAC 地址为什么是 `ff:ff:ff:ff:ff:ff`（广播地址）？
发送方不知道目标 IP 对应的 MAC 地址，只能通过广播让网段内所有设备接收请求，IP 匹配的设备才会回复，这是 ARP 协议的标准设计


2. 为什么 ARP 缓存中的条目会在几分钟后自动删除？
为了适配网络拓扑变化，避免使用过时的 IP-MAC 映射，同时释放缓存资源，这是 ARP 缓存的基本工作机制


3. 如果 ARP 缓存中的 MAC 地址已经过期（对方 IP 对应的设备已更换），会出现什么问题？如何解决？
过期会导致主机向错误的 MAC 发送数据，通信中断；解决方法包括手动删除缓存（arp -d）或等待超时刷新

简答题：

1. `ping` 命令发送的是什么类型的 ICMP 消息？收到的回复又是什么类型？
ping 命令发送的是 ICMP Echo Request（类型 8）消息，收到的回复是 ICMP Echo Reply（类型 0）消息


2. 为什么 ping 不同目标的 TTL 值不同？TTL 值反映了什么信息？
不同目标的 TTL 值不同，是因为数据包经过的路由器跳数不同，每经过一台路由器 TTL 会减 1。TTL 值反映了数据包从源到目标经过的网络跳数，也间接体现了目标设备的操作系统类型和距离远近


3. 教材表 2.4 中列出了多种 ICMP 消息类型。`Destination unreachable`（类型 3）在什么情况下会出现？请用以下方法尝试触发并观察：

   ```bash

   # 例如你的本机 IP 是 192.168.1.100，子网掩码 255.255.255.0，
   # 那么可以 ping 192.168.1.250（一个大概率没有被分配的地址）
   ```

## 问答题

1. 网络包由哪几部分构成？IP 头部和 MAC 头部分别的作用是什么？
网络包由 以太网帧头部（MAC 头部）、IP 头部、传输层头部（TCP/UDP）、应用层数据、FCS 校验尾 构成。IP 头部负责跨网段寻址，记录源 / 目的 IP、TTL、协议类型等信息，实现端到端路由；MAC 头部负责局域网内链路层传输，记录源 / 目的 MAC 地址和以太类型，实现同一网段内设备间的帧交付


2. IP 协议和以太网协议在网络传输中分别承担什么职责？它们是如何分工协作的？
IP 协议（网络层）负责跨网段的逻辑寻址、路由选择和分片重组，让数据能从源主机路由到目标主机；以太网协议（数据链路层）负责同一网段内的物理帧传输、差错检测和介质访问控制。二者分工协作：IP 地址标识跨网段的通信端点，MAC 地址标识单网段内的转发节点，数据在不同网段传输时，IP 地址保持不变，MAC 地址随链路跳数更新，共同完成从源到目标的通信


3. ARP 协议解决的核心问题是什么？如果不使用 ARP 缓存，网络中会出现什么情况？
ARP 协议解决的核心问题是IP 地址与 MAC 地址的映射解析，即根据目标 IP 地址获取对应的 MAC 地址，实现局域网内数据帧的正确交付。如果不使用 ARP 缓存，每次通信都需要广播发送 ARP 请求，会导致网络中 ARP 广播包泛滥，增加带宽占用和设备处理开销，降低通信效率


4. 为什么 IP 和负责传输的网络（如以太网）要分开设计？这种设计带来了什么好处？
分开设计是为了实现网络层与数据链路层的解耦。IP 协议不依赖底层传输网络的具体类型（以太网、Wi-Fi、PPP 等），底层传输网络也无需感知上层 IP 协议的细节。这种设计的好处是提高了网络的灵活性和可扩展性，IP 协议可以在不同类型的物理网络上运行，底层传输网络技术升级也不会影响 IP 层的工作


5. 网卡在发送包时会额外添加哪 3 个控制数据？它们各自的作用是什么？
网卡发送包时会额外添加：① 前导码（Preamble）：用于接收方同步时钟，标识帧的开始；② 帧起始定界符（SFD）：标志前导码结束，帧数据开始；③ 帧校验序列（FCS）：基于帧数据计算的校验值，用于接收方检测传输过程中的数据错误。


6. 网卡接收到一个包后，需要经过哪些步骤才能将其交给操作系统？如果 FCS 校验失败会怎样？
网卡接收包后，步骤为：①检测前导码和 SFD，同步时钟；②验证目的 MAC 地址是否匹配本机；③校验 FCS，检查数据是否损坏；④剥除以太网帧头部和尾部，将 IP 数据报交给操作系统。如果 FCS 校验失败，网卡会直接丢弃该帧，不向上层传递，避免错误数据进入协议栈。



7. TCP 和 UDP 的核心区别是什么？请从连接管理、可靠性、效率、适用场景四个维度进行比较。
连接管理：TCP 是面向连接的协议，通信前需三次握手建立连接，通信后四次挥手释放连接；UDP 是无连接协议，无需建立连接，直接发送数据。
可靠性：TCP 提供可靠交付，通过确认、重传、序号机制保证数据不丢失、不重复、按序到达；UDP 不保证可靠交付，无确认和重传机制，数据可能丢失、乱序。
效率：TCP 头部开销大（至少 20 字节），且有流量控制、拥塞控制机制，传输效率较低；UDP 头部仅 8 字节，无额外控制机制，传输效率高、延迟低。
适用场景：TCP 适用于文件传输、网页访问等对可靠性要求高、可容忍延迟的场景；UDP 适用于视频通话、游戏直播、实时语音等对实时性要求高、可容忍少量丢包的场景。


8. UDP 适用于哪些场景？请结合教材内容解释为什么这些场景适合使用 UDP 而非 TCP。
UDP 适用于视频通话、在线游戏、实时直播、DNS 查询等场景。这些场景对实时性要求极高，少量丢包对用户体验影响不大，但 TCP 的连接建立、确认重传和拥塞控制会带来额外延迟，影响实时性；UDP 无连接、低延迟、开销小的特点更契合这类场景的需求，同时部分应用会在应用层实现自定义的差错恢复机制，弥补 UDP 可靠性不足的问题。


9. 如果一个 IP 包经过多次路由转发后 TTL 降为 0，路由器会如何处理？这与教材中提到的哪种 ICMP 消息有关？
当 IP 包的 TTL 降为 0 时，路由器会丢弃该数据包，并向源主机发送 ICMP Time Exceeded（类型 11，超时） 消息，告知数据包因 TTL 耗尽无法到达目标主机，避免数据包在网络中无限循环转发。
