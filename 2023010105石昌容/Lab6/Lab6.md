# Lab6：第一次写防火墙规则，做基础包过滤

## 实验背景

前面的实验已经让你看到一次网络通信里的 IP 地址、端口号、协议类型和连接状态。

从本次实验开始，我们正式进入防火墙规则配置。防火墙最基础的能力是**包过滤**：根据数据包里的字段决定放行还是拦截。

本实验重点观察以下内容：

1. 防火墙规则如何匹配协议、源地址和目的端口
2. `INPUT` 链为什么会影响别人访问本机服务
3. `ACCEPT`、`REJECT`、`DROP` 三种动作的现象差异
4. 规则顺序为什么会影响最终访问结果
5. 规则计数器如何帮助判断规则是否被命中
6. 为什么基础包过滤看的是 IP、端口、协议，而不是 HTTP 内容

> **环境说明**：推荐在 Ubuntu 虚拟机、WSL2 或其他 Linux 实验环境中完成。  
> macOS 和 Windows 本机没有相同的 `iptables` 实验环境。  
> 不要在真实服务器或已有安全策略的主机上随意清空防火墙规则。

### 任务二：添加 REJECT 规则并观察访问失败

**第一步：在终端 C 插入一条拒绝规则**

```bash
sudo iptables -I INPUT 1 -p tcp --dport 38060 -j REJECT
```

命令说明：

| 部分 | 含义 |
| :--- | :--- |
| `sudo` | 以管理员权限执行 |
| `iptables` | 修改 Linux 防火墙规则 |
| `-I INPUT 1` | insert，把规则插入到 `INPUT` 链第 1 行 |
| `-p tcp` | protocol，只匹配 TCP 协议 |
| `--dport 38060` | destination port，只匹配目的端口为 `38060` 的 TCP 包 |
| `-j REJECT` | jump，命中规则后执行 `REJECT`，拒绝并给对方返回错误 |

常见可选写法：

| 写法 | 作用 |
| :--- | :--- |
| `-A INPUT ...` | append，把规则追加到链尾；和 `-I` 相比，顺序可能不同 |
| `-s 192.168.1.10` | 只匹配某个源 IP |
| `-s 192.168.1.0/24` | 只匹配某个源网段 |
| `-d 127.0.0.1` | 只匹配某个目的 IP |
| `--sport 12345` | 匹配源端口，通常不如目的端口常用 |
| `-i lo` | 只匹配从指定入接口进入的流量，例如回环接口 `lo` |
| `-j ACCEPT` | 命中后放行 |
| `-j DROP` | 命中后直接丢弃，不回应 |
| `-j LOG` | 命中后写日志，通常还需要后续规则继续处理 |
| `-j REJECT --reject-with tcp-reset` | 对 TCP 连接用 RST 明确拒绝 |

这条规则的含义是：凡是进入本机、协议是 TCP、目的端口是 `38060` 的包，都拒绝。

简答题：

1. 这条规则主要匹配了哪些字段？
答：协议字段：-p tcp，指定只匹配 TCP 协议的数据包。
目标端口字段：--dport 38060，指定只匹配目标端口为 38060 的数据包。
流量方向字段：INPUT 链，指定只匹配进入本机的流量。
此外，源地址和目标地址默认是 0.0.0.0/0，即匹配所有地址，无需额外指定。


2. 为什么服务端没有收到请求，也能说明防火墙已经在更前面拦截了流量？
答：数据包处理顺序：在 Linux 系统中，iptables 防火墙工作在内核态，位于用户态服务端程序的前面。
当客户端发起连接时，数据包会先经过内核的 iptables 规则链处理，再交给用户态的服务端程序。
拦截的直接证据：如果防火墙规则匹配并拦截了数据包，内核会直接处理（例如 REJECT 会返回拒绝包），不会将数据包向上传递给用户态的服务端进程。
因此，服务端日志没有收到任何请求记录，就说明请求在到达服务端之前，就被 iptables 在内核层拦截了。


3. `REJECT` 和“服务端程序没启动”在客户端现象上可能有什么相似之处？
答：两者在客户端现象上的核心相似点是：客户端都会收到 Connection refused (Errno 111) 错误，而不是超时等待。
REJECT 场景：防火墙主动向客户端返回一个 ICMP port unreachable 或 TCP RST 包，客户端会立即收到拒绝响应。
服务端未启动场景：操作系统内核会检测到目标端口没有进程监听，同样会主动向客户端返回 RST 包，客户端也会立即收到拒绝响应。
因此，仅从客户端现象上看，这两种情况都表现为 “连接被拒绝”，很难直接区分。

### 任务三：添加 ACCEPT 规则并观察规则顺序

现在 `REJECT` 规则已经阻断了访问。接下来在它前面插入一条更具体的允许规则。

**第一步：在终端 C 插入允许规则**

```bash
sudo iptables -I INPUT 1 -p tcp -s 127.0.0.1 --dport 38060 -j ACCEPT
```

命令说明：

| 部分 | 含义 |
| :--- | :--- |
| `-I INPUT 1` | 插入到 `INPUT` 链第 1 行，让这条规则排在原来的 `REJECT` 前面 |
| `-p tcp` | 匹配 TCP 协议 |
| `-s 127.0.0.1` | source，只匹配源地址为 `127.0.0.1` 的流量 |
| `--dport 38060` | 只匹配目的端口为 `38060` 的流量 |
| `-j ACCEPT` | 命中后放行 |

常见可选写法：

| 写法 | 作用 |
| :--- | :--- |
| `-s 10.0.0.0/24` | 允许一个网段 |
| `-s 10.0.0.5` | 只允许一台主机 |
| `-d 10.0.0.10` | 限定目的地址 |
| `-i eth0` | 限定从某个网卡进入 |
| `-m comment --comment "allow lab6"` | 给规则加注释，便于排错和维护 |

这条规则的含义是：允许源地址为 `127.0.0.1`、目的端口为 `38060` 的 TCP 访问。

**第二步：查看规则顺序**


具体显示格式可能略有不同，但关键是 `ACCEPT` 在 `REJECT` 前面。

简答题：

1. 为什么同样存在 `REJECT` 规则，访问却恢复了？
答：因为在 REJECT 规则的前面，添加了一条更优先匹配的 ACCEPT 规则：sudo iptables -I INPUT 1 -p tcp -s 127.0.0.1 --dport 38060 -j ACCEPT
这条规则会匹配你客户端（源地址 127.0.0.1）访问目标端口 38060 的请求。
当请求到达时，会先匹配到这条 ACCEPT 规则，直接被放行，不会再继续往下匹配后面的 REJECT 规则。

2. 为什么防火墙规则顺序会影响最终结果？
答：iptables 是按顺序匹配、匹配即停止的：
内核会从上到下依次检查 INPUT 链里的规则。
一旦数据包匹配到某条规则（比如 ACCEPT 或 REJECT），就会执行该规则的动作，并停止后续规则的匹配。
因此，先匹配到的规则决定了数据包的命运，后面的规则对它不再生效。
这也是为什么同样的规则，调换顺序后结果会完全相反。


3. 如果把 `REJECT` 放在 `ACCEPT` 前面，本次访问会发生什么？
答：客户端的请求会先匹配到 REJECT 规则，被直接拒绝。
客户端会收到 Connection refused 错误，无法建立连接。
后面的 ACCEPT 规则永远不会被匹配到，请求依然无法访问服务端。

### 任务四：对比 DROP 与 REJECT

`REJECT` 会明确拒绝访问，`DROP` 则是直接丢弃包，不给对方回应。下面观察两者在客户端现象上的差异。

**第一步：删除刚才添加的 ACCEPT 和 REJECT 规则**

```bash
sudo iptables -D INPUT -p tcp -s 127.0.0.1 --dport 38060 -j ACCEPT
sudo iptables -D INPUT -p tcp --dport 38060 -j REJECT
```

命令说明：

| 部分 | 含义 |
| :--- | :--- |
| `-D INPUT` | delete，从 `INPUT` 链删除一条规则 |
| 后面的匹配条件 | 必须和要删除的规则一致，否则可能提示规则不存在 |

也可以按行号删除，例如：

```bash
sudo iptables -D INPUT 1
```

按行号删除前一定要先执行 `sudo iptables -L INPUT -n --line-numbers` 确认行号，因为删除一条规则后，后面的行号会立刻变化。

删除后查看确认：

```bash
sudo iptables -L INPUT -n -v --line-numbers
```

**第二步：添加 DROP 规则**

```bash
sudo iptables -I INPUT 1 -p tcp --dport 38060 -j DROP
```

命令说明：

| 部分 | 含义 |
| :--- | :--- |
| `-I INPUT 1` | 插入到 `INPUT` 链第 1 行 |
| `-p tcp --dport 38060` | 匹配访问本机 `38060` 端口的 TCP 流量 |
| `-j DROP` | 命中后直接丢弃，不返回拒绝信息 |

**第三步：用 5 秒超时运行客户端**

命令说明：

| `python3 firewall_lab6_client.py` | 运行客户端脚本 |

这里设置较短超时，是为了观察 `DROP` 造成的等待现象，同时避免客户端一直卡住。

观察客户端是否等待一段时间后才失败。再观察服务端是否收到请求。

简答题：

1. `REJECT` 和 `DROP` 都能阻断访问，为什么客户端看到的现象不同？
答：两者的核心区别在于内核是否向客户端返回响应：
REJECT：内核会主动向客户端返回一个拒绝包（如 ICMP port unreachable 或 TCP RST），客户端会立即收到 Connection refused 错误，几乎没有等待时间。
DROP：内核会直接静默丢弃数据包，不返回任何响应。客户端不知道请求被拦截，只能一直等待，直到超时后才会报错（如你实验中的 timeout after 5.006s）。


2. 如果你是网络管理员，排错时哪一种动作更容易判断问题？为什么？
答：客户端会立即收到明确的错误提示（Connection refused），可以快速定位到 “目标端口被防火墙主动拒绝”。
而 DROP 只会导致超时，客户端无法区分是 “防火墙拦截”“网络不通” 还是 “服务端宕机”，排错时需要额外排查链路、服务状态等多种可能性，效率更低。


3. 如果你是攻击者，`DROP` 可能会让扫描结果变得更不明确，原因是什么？
答：攻击者扫描端口时，主要通过响应判断端口状态：
如果目标使用 REJECT，扫描工具会收到明确的拒绝响应，从而直接判断端口 “关闭”。
如果目标使用 DROP，扫描工具只会收到超时，无法区分端口是 “被防火墙静默拦截”，还是 “主机不存在 / 端口未开放 / 网络被过滤”。
这种模糊性会让攻击者难以准确判断目标的网络拓扑和端口状态，增加了扫描的不确定性。

## 问答题

1. 本实验中的包过滤规则主要匹配了哪些字段？
答：协议字段：-p tcp，只匹配 TCP 协议的数据包。
目标端口字段：--dport 38060，匹配访问目标端口 38060 的请求。
源地址字段：部分规则使用了-s 127.0.0.1，只匹配来自本机的请求。
流量方向：通过INPUT链，只匹配进入本机的流量。



2. 为什么说本实验实现的是“包过滤”，而不是“应用层代理”？
答：包过滤（iptables）：工作在内核的网络层 / 传输层，只检查 IP、端口、协议等数据包头部信息，不解析 HTTP 等应用层内容，处理效率高，不与应用层进程交互。
应用层代理：工作在用户态，需要解析完整的应用层协议（如 HTTP 请求头、URL），并由代理进程中转请求，会带来额外开销。
本实验中，规则直接在系统内核拦截数据包，没有任何应用层进程参与，因此是包过滤。


3. `INPUT`、`OUTPUT`、`FORWARD` 分别对应什么方向的流量？
答：INPUT：目标地址是本机的流量（外部 / 本机进程发给本机的数据包）。
OUTPUT：源地址是本机的流量（本机进程发给外部的数据包）。
FORWARD：流经本机但不是发给本机、也不是本机发出的流量（本机作为路由器转发的数据包）。


4. 为什么本实验主要操作 `INPUT` 链，而不是 `FORWARD` 链？
答：本实验是访问本机上的服务端，客户端和服务端都在同一台机器上，数据包的目标地址是本机，只会经过INPUT链，不会经过FORWARD链。FORWARD链只有在本机作为网关 / 路由器转发流量时才会用到，因此本次实验无需操作它。


5. `ACCEPT`、`REJECT`、`DROP` 三种动作的区别是什么？
答:ACCEPT：表示允许数据包通过，流量会正常转发给服务端，客户端可以成功建立连接并收到响应。
REJECT：表示主动拒绝数据包，并向客户端返回拒绝信息，客户端会立即收到连接被拒绝的错误提示，没有等待时间。
DROP：表示直接静默丢弃数据包，不向客户端返回任何信息，客户端会一直等待直到超时后才报错，等待时间明显更长。


6. 为什么规则顺序会影响最终结果？
答：iptables 是按顺序匹配、匹配即停止的：内核会从上到下依次检查规则，一旦数据包匹配到某条规则（如ACCEPT），就会执行该动作并停止后续规则的匹配。
如果ACCEPT规则在前面，请求会被直接放行，后面的REJECT规则永远不会被匹配到。
如果REJECT规则在前面，请求会被直接拒绝，后面的ACCEPT规则也不会生效。


7. 规则计数器在排错时有什么用？
答：规则计数器（pkts和bytes）会记录该规则匹配到的数据包数量和字节数，是判断规则是否生效的直接证据：
如果请求被拦截，但计数器没增加，说明请求没有匹配到这条规则，可能是规则顺序、匹配条件写错了。
如果计数器增加了，说明规则成功命中了请求，证明规则本身是有效的。


8. 真实环境中为什么常常采用“默认拒绝，再按需放行”的策略？
答：这种策略遵循最小权限原则：
默认拒绝所有流量，只开放业务必需的端口 / 地址，能最大限度减少攻击面，避免因规则疏漏导致的意外暴露。
相比 “默认允许，再按需拒绝”，这种策略更安全，不会因为忘记添加拒绝规则而导致端口被非法访问。


9. 只靠本实验这种基础包过滤规则，还无法解决哪些更复杂的安全问题？
答：无法解析应用层协议：无法识别 HTTP 请求中的恶意 URL、SQL 注入、XSS 攻击等，也无法基于 URL、Cookie 等应用层信息做访问控制。
无法处理动态端口 / 复杂协议：如 FTP、SIP 等使用动态端口的协议，仅靠静态端口规则无法控制。
无法抵御部分攻击：如 SYN Flood 等 DoS 攻击，基础包过滤无法识别异常流量特征，需要结合状态检测或专用防护工具。
无法进行身份验证：无法基于用户、Token 等身份信息进行访问控制，只能控制 IP 和端口。
